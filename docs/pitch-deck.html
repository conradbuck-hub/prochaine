Prochaine

A personal, bilingual transit chatbot for Montreal — one chat window covering STM métro and bus, REM, exo, BIXI, and weather. Installed as a PWA on a phone. Built for one person, to be at a bus stop in the cold and just ask.

02 / 16
Problem

Five apps to leave the house

STM's app for métro and bus, Transit or exo's app for commuter rail, the BIXI app for bikes, a weather app to know whether any of it matters today — and none of them speak the way you actually talk.

"sherbrook atwatter c'est quoi le prochain" shouldn't need a perfectly-typed, perfectly-accented query to get an answer.
03 / 16
Solution

One chat window

Ask in whatever register, spelling, or language you'd actually use. Get back one clean answer, in one language, sourced from real feeds — not a bot's best guess.

04 / 16
Core rule

The LLM never invents a schedule

The model has exactly two jobs: parse the question, phrase the answer. Every departure time, delay, or alert comes from code querying a real feed. If a feed is down, the answer says so.

Voice & text

Comprehension adapts. Voice never does.

Any register, spelling, or language in — clean, neutral Quebec French or plain English out, one language per reply, never mixed.

« 3 min · 24 vers Côte-Vertu. Prochain dans 11 min. »
"3 min · 24 toward Côte-Vertu. Next in 11 min."
« Je comprends, c'est frustrant. Le 105 est retardé de 8 min — voici le prochain départ fiable. »

13 / 16
Cost

~$5/month at one user

Single Railway instance, one LLM call pair per query, JSON-file persistence. The architecture is intentionally cheap because it only has to be right for one person — scaling changes almost every assumption here.

14 / 16
Audience

Who else might want this

  • Households sharing common commutes (not just one user's stops)
  • Small orgs wanting a Slack/SMS transit assistant for a specific site
  • Other transit-agency cities with public GTFS + GTFS-RT feeds
15 / 16
Risks & moat

The moat isn't the feed integration

GTFS parsing is commodity work. The actual differentiator is the voice doctrine — disciplined, correctly-registered bilingual output that never sounds like a translated FAQ — and never inventing a schedule. Both are easy to erode with a careless prompt change.

16 / 16
Path to more users

What would actually have to change

  • Multi-tenant profile store (today: one JSON file, one user)
  • Per-user auth instead of a single trusted device
  • Rate limiting and cost controls per tenant on the LLM calls
  • A real onboarding UI beyond "the first two chat messages"

None of this is started, and none of it should be, before Stage 4's failure log says the single-user product actually works.
